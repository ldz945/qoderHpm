# 枚举与状态机定义

本文档定义 HPM 项目管理系统中的所有状态机和枚举值。

---

## 一、项目状态机（Project Status）

### 1.1 状态定义

| 状态代码 | 中文名 | 触发条件 | 控制范围 | 颜色 |
|---------|--------|---------|---------|------|
| draft | 草稿 | 第一版计划未发布前 | 1. 上级任务开始/结束时间可从下往上汇总（每次保存触发）；其他状态下上级限制下级。<br>2. 项目经理授权的负责人可增加/更改/删除下级任务和资源；项目经理有全部权限。<br>3. 不保存版本 | - |
| approved | 批准中 | 项目计划提交发布时触发 | 不可更改计划或填写进度；审批拒绝返回提交前状态 | - |
| planning | 计划 | 项目计划审批完成 | 过渡状态，不可修改计划不可填写进度 | - |
| ongoing | 进行中 | 两种方式：<br>1. 第一版计划发布时项目生效日期 < 当前时间自动变更；<br>2. 项目生效日期 > 当前时间则保持草稿，后续手动更改 | 可进行进度填写和交付物上传 | 红/黄/绿(根据预警) |
| hold | 暂挂 | 进行状态下项目经理手工暂挂，需 PL 审批 | 不允许计划变更，不允许进度填写 | - |
| terminal | 终止 | 进行/暂挂状态下手工更改，需 PL 审批 | 可进行项目验收；终止后可逆 | - |
| cancel | 取消 | 进行/暂挂状态下手工更改，需 PL 审批 | 不进行验收；取消不可逆；已填报工时依旧有效 | - |
| modife | 可修改 | 三种方式：<br>1. 项目经理直接更改(不用审批)；<br>2. 变更管理平台触发；<br>3. 仅进行中状态可更改 | 不可填写进度；项目经理可更改所有计划；授权的任务负责人可更改下级 | - |
| pre-closure | 预关闭 | 进行中状态手动更改 | 可进行项目验收 | - |
| closure | 关闭 | 项目验收完成后手工关闭；重新打开需 PL 审批 | 计划和文档仅可查看；进度不可填写 | - |

### 1.2 状态转换图

```
                    ┌─────────────────────────────────────────┐
                    │                                         │
                    ▼                                         │
draft ──► approved ──► planning ──► ongoing ──► hold ─────────┘
                                      │          │
                                      │          ▼
                                      │      ongoing (恢复)
                                      │
                                      ├─► terminal (可逆)
                                      │
                                      ├─► cancel (不可逆)
                                      │
                                      ├─► modife ──► ongoing (修改完成后)
                                      │
                                      └─► pre-closure ──► closure
```

### 1.3 状态流转说明

| 源状态 | 目标状态 | 触发条件 | 备注 |
|--------|---------|---------|------|
| draft | approved | 提交项目计划发布 | - |
| approved | draft | 审批拒绝 | 返回提交前状态 |
| approved | planning | 审批通过 | 过渡状态 |
| planning | ongoing | 项目生效日期 < 当前时间 | 自动变更 |
| ongoing | hold | 项目经理手工暂挂 | 需 PL 审批 |
| hold | ongoing | 恢复项目 | - |
| ongoing | terminal | 手工更改 | 需 PL 审批，可逆 |
| ongoing | cancel | 手工更改 | 需 PL 审批，不可逆 |
| ongoing | modife | 项目经理更改/变更平台触发 | 仅进行中状态可更改 |
| modife | ongoing | 修改完成 | - |
| ongoing | pre-closure | 手动更改 | - |
| pre-closure | closure | 验收完成 | - |
| closure | ongoing | 重新打开 | 需 PL 审批 |

## 二、任务状态机（Task Status）

### 2.1 状态定义

| 状态代码 | 中文名 | 说明 | 备注 |
|---------|--------|------|------|
| draft | 草稿 | 项目为草稿状态时所有任务为草稿；项目已发布则未发布阶段任务为草稿 | - |
| approved | 批准中 | 授权任务负责人取消任务需项目经理审批，未通过前为批准中 | 不可填写进度 |
| planning | 计划 | 已发布的任务状态 | - |
| ongoing | 进行中 | 第一级任务计划开始时间 < 当前时间，自动变更；或界面点击"任务状态更新"触发 | - |
| modife | 可修改 | 仅进行中状态可更改为可修改 | 任务在草稿/计划/进行中可修改 |
| cancel | 取消 | 项目经理取消不触发审批；授权负责人取消需项目经理审批 | 取消不可逆；已填报工时有效 |
| closure | 关闭 | 进度 100% 后手动关闭；父任务关闭子任务联动；进度非 100% 关闭时备注必输 | 关闭后可再打开 |

### 2.2 状态流转图

```
draft ──► planning ──► ongoing ──► closure
            │            │
            │            ├─► modife ──► ongoing
            │            │
            │            └─► cancel (不可逆)
            │
            └─► approved (取消审批中)
```

### 2.3 状态流转说明

| 源状态 | 目标状态 | 触发条件 | 备注 |
|--------|---------|---------|------|
| draft | planning | 项目发布/阶段发布 | - |
| planning | ongoing | 计划开始时间 < 当前时间 或 手动触发 | - |
| ongoing | modife | 项目进入可修改状态 | - |
| modife | ongoing | 修改完成 | - |
| ongoing | cancel | 项目经理取消 或 授权负责人申请取消 | 后者需审批 |
| ongoing | closure | 进度 100% 手动关闭 | 父任务关闭子任务联动 |
| closure | ongoing | 重新打开 | - |

## 三、变更单状态（Change Order Status）

### 3.1 状态定义

| 状态代码 | 中文名 | 触发条件 | 说明 |
|---------|--------|---------|------|
| new | 新建 | 创建变更单 | 初始状态 |
| approving | 审批中 | 提交审批 | - |
| rejected | 驳回 | 审批人驳回 | 可修改后重新提交或手动关闭 |
| approved | 批准 | 审批通过 | 关键路径变更：对应任务状态变为可修改 |
| closed | 关闭 | 手动关闭 | 终态 |

### 3.2 状态流转图

```
new ──► approving ──► approved
  │         │
  │         ▼
  │      rejected ──► approving (重新提交)
  │         │
  │         ▼
  └────► closed (手动关闭)
```

## 四、验收单状态（Acceptance Status）

### 4.1 状态定义

| 状态代码 | 中文名 | 触发条件 | 说明 |

| 代码值 | 中文名 | 详细行为说明 |

| SS | 开始-开始 | 后续任务开始依赖紧前任务开始；紧前开始变动导致后续开始变动；后续开始不能小于紧前开始 |
